
# 📊 SQL JOIN Interview Question: Number of Records with Duplicate Keys
```sql
USE em;

CREATE TABLE join_tabl1 (id1 INT);
CREATE TABLE join_tabl2 (id2 INT);

-- Insert duplicate values into Table 1
INSERT INTO join_tabl1 VALUES (1);
INSERT INTO join_tabl1 VALUES (1);

-- Insert duplicate values into Table 2
INSERT INTO join_tabl2 VALUES (1);
INSERT INTO join_tabl2 VALUES (1);
INSERT INTO join_tabl2 VALUES (1);

SELECT * FROM join_tabl1; -- 2 records
SELECT * FROM join_tabl2; -- 3 records

```
## 🔗 How JOIN Works with Duplicate Keys
When joining on duplicate keys:
- Each matching row in the left table pairs with **every matching row** in the right table.
- Total rows = **(count in left) × (count in right)**
- Here: `2 × 3 = 6 rows`

## 🔸 INNER JOIN
```sql
SELECT *
FROM join_tabl1
INNER JOIN join_tabl2
ON join_tabl1.id1 = join_tabl2.id2;
```

## 🔸 LEFT JOIN
```sql
SELECT *
FROM join_tabl1
LEFT JOIN join_tabl2
ON join_tabl1.id1 = join_tabl2.id2;
```

## 🔸 RIGHT JOIN
```sql
SELECT *
FROM join_tabl1
RIGHT JOIN join_tabl2
ON join_tabl1.id1 = join_tabl2.id2;
```

## 🔸 FULL OUTER JOIN *(SQL Server / PostgreSQL)*
```sql
SELECT *
FROM join_tabl1
FULL OUTER JOIN join_tabl2
ON join_tabl1.id1 = join_tabl2.id2;
```

* Duplicate keys create a **Cartesian multiplication effect** for matching values.
* If all values match:
  * `INNER JOIN`, `LEFT JOIN`, `RIGHT JOIN`, and `FULL OUTER JOIN` → **same result**
- If unmatched rows exist:
- JOIN types start to differ in output
